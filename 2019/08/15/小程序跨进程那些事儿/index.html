小程序跨进程那些事儿

发表于 2019-08-15 | 分类于 Android |

本文系个人原创，禁止任何形式的转载。欢迎评论发表你的看法。

众所周知，小程序在Android上是独立进程的，这样做一方面有利于多个小程序之间运行环境隔离，同时在操作系统层面单个小程序类似一个独立的App；另一方面进程之间可循环复用，支持同时打开多个小程序。但是独立进程后必然会涉及很多跨进程调用和跨进程通信的场景，例如获取账号信息/登陆相关，分享，插件, 读写sp，进程保活等等，某些场景下考虑到页面栈问题也必须跨进程调用，那么小程序是如何进行跨进程调用的呢？

跨进程调用

ContentProvider跨进程调用

对于同步并且无需context的跨进程调用，通过在主进程预埋一个ContentProvider，指定其authorities，然后为了方便调用，我们提供了一个抽象类Delegation，抽象方法exec供子类重写执行相应逻辑。调用方只需要通过传入一个context和一个用于执行具体逻辑的Delegation子类即可将逻辑放在主进程执行，并同步返回。原理就是利用ContentResolver的call方法来执行ContentProvider的call方法，并传入delegation class name，ContentProvider的call方法利用反射创建delegation实例，调用exec方法，并将结果返回。这种调用方式较为轻量，解决sp跨进程读写问题亦可采用这种方式。

activity代理

对于需要context的异步跨进程调用（需不需要Context一方面看调用的业务，另一方面要考虑到页面栈问题，这个后面细讲），通过在主进程预埋一个standard的透明代理activity，用于执行delegation业务逻辑。通过startActivity调起代理activity，代理activity onCreate里从intent取出代理class name和参数，反射创建代理实例执行相应逻辑，当异步过程结束后将结果通过setResult返回给原调用activity的onResult，finish代理activity。

messenger

messenger在进程间通信是常用手段，在小程序activity创建之后会bind一个service，service建立和主进程的连接，通过messenger通信，在msg中传入delegation class name和type进行IPC调用。

DeathRecipient

跨进程通信过程中需要利用DeathRecipient保证连接的可靠性，通过linkToDeath方法传入一个DeathRecipient，在remote binder因为进程被kill而消失的时候会调用DeathRecipient的binderDied()方法，利用这个我们可以监听到进程被kill。对于小程序进程来讲，主进程被kill会尝试唤醒主进程，重新建立连接。对主进程来说，某个小程序进程被kill后会断开连接，然后预创建小程序进程，重新建立连接。

跨进程读写sp

sp读除了首次会从文件同步到内存后面都是基于内存，写分同步异步，apply会先将内容写入内存，再异步写入文件，commit则是同步写入文件。虽然本质是基于文件，但是由于依赖内存去读，所以sp在多进程下会存在问题。目前读取通用配置等sp使用ContentProvider代理到主进程读，从而规避进程间同步问题。关于sp的原理推荐：http://gityuan.com/2017/06/18/SharedPreferences/

插件页面栈

百度app使用的插件众多，由于百度app插件框架的限制，小程序进程无法直接用自身的context来调起插件页面，在小程序进程中直接调起插件其实最终插件框架使用的是application的context来调起的插件。而且插件大多launchMode是“singleTask”，taskAffinity未明确指定，这就导致小程序中直接调起插件，插件页面会进入百度app页面栈，这里再简单介绍下launchMode和页面栈的关系。

launchMode分：”standard”, “singleTop”, “singleTask”, “singleInstance”。”standard”和”singleTop”相似，都是可创建多个实例，但不同的是singleTop在页面栈顶已经存在实例的情况下会进行复用，回调onNewIntent。而且这俩不论taskAffinity如何都会进入启动他的activity的页面栈。”singleTask”和”singleInstance”相似，只能有一个实例。”singleTask”相当于是自带FLAG_ACTIVITY_CLEAR_TOP标志，栈内复用时会pop栈顶的activity。这俩差别在于”singleInstance”所在栈内只允许这一个activity存在，使用”singleInstance”的activity去启动另一个activity的话intent就像自动带上了FLAG_ACTIVITY_NEW_TASK标志。页面栈的话singleTask受自身的taskAffinity影响，如果和启动他的activity（不为singleInstance）所属栈 taskAffinity相同，会进入启动它的taskAffinity页面栈中，如果不同，则不在同一页面栈。”singleInstance”则不受taskAffinity影响，独处一栈。

当然上面讨论的都是activity启动activity的情况，事实上，其他context也能启动activity，例如service，application。但是context（特指非activity context）启动activity的时候必须加上FLAG_ACTIVITY_NEW_TASK标志，否则直接会崩溃，因为被启动activity无法知晓自己应该存在于哪个栈中。具体可见ContextImpl的源码：

这种情形下启动activity，activity归属栈和taskAffinity直接相关。也就是什么taskAffinity进什么栈(singleInstance依旧例外)。

回到之前讨论的小程序进程直接调起插件，由于使用的是application context调起，taskAffinity又没指定，所以插件activity运行在主进程页面栈，导致回退时候直接退回百度app了。因为插件调用来源不止小程序，而且插件多，改插件的taskAffinity虽能解决问题，但是实属下下策。所以多数场景我们采用activity跨进程代理调用插件，这样相当于代理activity运行在主进程，但是在小程序页面栈，同时代理activity调起插件，只要插件activity是standard的，就会归属于小程序页面栈，从而解决页面栈回退问题。
